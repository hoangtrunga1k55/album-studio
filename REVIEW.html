Code Review - Album Studio - 2026-07-04

Scope: full app review for handoff, speed, and stability. Stack: Tauri 2 + Rust backend + React/TypeScript + Konva.

Summary. App structure is understandable and the current build/test baseline passes, but the export and import pipelines are not yet safe for large real albums. The main handoff risk is memory pressure from base64-heavy rendering/caching and high-frequency per-image state updates.
Findings: 0 CRITICAL · 3 HIGH · 5 MEDIUM · 2 LOW

HIGH

[H1] Export keeps every rendered spread and final file in memory before writing

File: app/src/engine/exportAlbum.ts:50, app/src/engine/exportAlbum.ts:60, app/src-tauri/src/export.rs:18

Issue: Export renders every spread into a JPEG data URL, stores all pages in pages, builds another files array of base64 payloads, then sends the whole payload over IPC for Rust to decode and write. For a 300 DPI album with many spreads, RAM can spike to multiple copies of each page: canvas pixels, JPEG bytes, base64 string, PDF embed bytes, IPC payload, and decoded Rust bytes. This is the top stability risk for handoff because real customer projects can freeze or crash during export.

Fix: Stream one spread at a time. Add a Tauri command like write_export_file(dir, name, b64) and write each JPG immediately after rendering. For PDF, either build incrementally while not retaining pages, or move PDF assembly to Rust/native code and pass file paths/bytes per page. Clear export image cache after export or expose an explicit eviction hook.

// safer shape
for (let i = 0; i < spreads.length; i++) {
  const page = await renderSpread(...);
  if (writeJpg) await writeExportFile(dir, `${prefix}${pad(i + 1)}.jpg`, stripDataUri(page.dataUrl));
  if (writePdf) await pdfBuilder.addJpg(stripDataUri(page.dataUrl), page.w, page.h);
  page.dataUrl = "";
}

[H2] Export/display image caches are unbounded base64 caches

File: app/src/ipc/export.ts:3, app/src/ipc/import.ts:47

Issue: getExportImage and getDisplayImage cache promises forever by path. Cached values are base64 data URIs, not lightweight handles. A studio session with hundreds of photos can retain many 1600px display JPEGs plus 3200px export JPEGs until the app is closed, even after images are removed from the album.

Fix: Replace both maps with a bounded LRU cache and provide clearDisplayImageCache(paths?)/clearExportImageCache(). Keep display cache small, and avoid caching export images globally unless the same asset is reused across many spreads.

[H3] Image import decodes all selected files in parallel and re-renders once per image

File: app/src-tauri/src/import.rs:88, app/src/components/ImagePanel.tsx:38, app/src/store/album.ts:202

Issue: Rust uses Rayon to process the entire file list concurrently, and React calls addImages([meta]) for every image event. This is responsive for small batches, but for 500-1000 large JPEG/HEIC files it can saturate CPU/RAM and trigger hundreds of Zustand updates and thumbnail grid re-renders.

Fix: Limit backend concurrency with a small pool based on CPU and memory profile, then batch UI events on the frontend. Accumulate image metas for 50-100 ms or 20-50 items and call addImages(batch) once per batch. Optionally add cancellation support for long imports.

MEDIUM

[M1] Export cancellation is only checked between spreads

File: app/src/engine/exportAlbum.ts:51, app/src/components/ExportDialog.tsx:123

Issue: The cancel flag is checked before each spread only. If one spread is slow due to HEIC decoding, high DPI canvas allocation, PDF embed, or IPC write, clicking "Hủy" does not interrupt that step.

Fix: Thread an AbortSignal or cancel token through renderSpread, image loading, PDF assembly, and write commands. At minimum, check cancellation after each image slot, after toDataURL, before PDF embed, and before each file write.

[M2] Hot render paths repeatedly search images by linear scan

File: app/src/engine/renderSpread.ts:63, app/src/components/SpreadCanvas.tsx:493

Issue: Each slot resolves images.find(...). Slot count is currently small, so this is not a blocker, but it is repeated in canvas render, sidebar/filmstrip, export, and property panels. As projects grow, this becomes avoidable work during interaction and export.

Fix: Derive an imageById map once per render/export and pass it down, or store a map in Zustand alongside the ordered list.

[M3] Frontend has no automated tests for critical album behavior

File: app/package.json:5

Issue: There is no frontend test script and no coverage for store operations such as auto-design, slot replacement, project open/save compatibility, transform persistence, and export option handling. Rust has one import pipeline test, but the React state layer holds most product logic.

Fix: Add Vitest tests for store/album.ts, engine/autoLayout.ts, template lookup, project serialization compatibility, and a smoke test for export orchestration with mocked IPC.

[M4] Build bundle is over Vite's 500 kB warning threshold

File: app/package.json:7, build output

Issue: pnpm build succeeds, but Vite reports dist/assets/index-*.js at about 1,016 kB minified / 362 kB gzip. For Tauri this is not as severe as a public web app, but it still affects first-window startup and memory.

Fix: Lazy-load export-only dependencies like pdf-lib and heavy panels/dialogs. Consider Rollup manual chunks for Konva/pdf code.

[M5] Saved project input is trusted without schema validation or migration defaults

File: app/src/App.tsx:124, app/src/store/album.ts:189

Issue: openProject parses arbitrary JSON and applies fields directly. A malformed or older .album file can set invalid currentIndex, missing arrays, unknown album size, or bad spread structures, causing blank UI or runtime errors after handoff.

Fix: Add a project schema validator and migration function keyed by version. Clamp currentIndex, default missing arrays/maps, and reject invalid album sizes/templates with a user-facing error.

LOW

[L1] Some recoverable failures are swallowed silently

File: app/src/App.tsx:57, app/src/App.tsx:67, app/src/engine/renderSpread.ts:54

Issue: Font/typo/template background failures are ignored. This is acceptable for best-effort UX, but for handoff/support it makes missing asset problems hard to diagnose.

Fix: Add lightweight structured logging or a diagnostics panel/export log so support can see which assets failed to load.

[L2] Default app metadata is still template-like

File: app/src-tauri/Cargo.toml:4, app/README.md:1

Issue: The Tauri package still says description = "A Tauri App" and authors = ["you"], and app/README.md is still the Vite/Tauri starter README. This is not runtime risk, but it weakens handoff quality.

Fix: Update product metadata and replace the app README with setup, build, release, runtime asset, and troubleshooting notes.

Passed

  • pnpm build passed on 2026-07-04.
  • cargo test passed: 1 Rust test, including JPEG/HEIC import pipeline over test-photos.
  • Tauri capability file is narrow: core, opener, and dialog only.
  • Import streams progress via Tauri channel instead of waiting for the whole folder.
  • Display rendering uses downscaled images instead of full-resolution originals.
  • Konva stages created for export are destroyed in finally.
  • Project save keeps image data out of the .album file and stores paths only, so project files remain lightweight.
